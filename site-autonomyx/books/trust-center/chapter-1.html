Identity Construct Model

Identity is the root construct for OpenAutonomyX governance.

Every infrastructure, platform, service, agent, workflow, tool, artifact, and decision must be bound to an identity before it can be trusted, authorized, audited, or scored.

Identity Spine

Linux user / group
→ LXD project / instance identity
→ OpenStack Keystone user / project / role
→ Kubernetes user / ServiceAccount / namespace
→ Juju user / model / application
→ Fabric DID / VC / account
→ Agent identity
→ Tool identity
→ Decision identity

Core Rule

No identity, no authority.

A construct may exist without trust, but it must not receive privileged capability without an identity binding and policy evaluation.

Identity Layers

| Layer | Identity Primitive | Governance Meaning |
|---|---|---|
| Linux | User, group, service account | Host-level authority |
| LXD | Project, profile, instance | Local machine boundary |
| OpenStack | Keystone user, project, domain, role | Cloud tenant authority |
| Kubernetes | User, group, ServiceAccount, namespace | Workload authority |
| Juju | User, controller, model, application | Operator authority |
| Fabric | DID, VC, account, relation | Cross-layer identity graph |
| Agentic OS | Agent, tool, workflow, decision | Governed work authority |

Identity Binding

Every construct should bind to:

  • Human owner
  • Organization owner
  • Runtime identity
  • Policy subject
  • Evidence subject
  • Trust score subject

Example: 

{
  "construct_id": "openstack.keystone.project.fabric-prod",
  "human_owner": "founder",
  "organization_owner": "openautonomyx",
  "runtime_identity": "keystone:project:fabric-prod",
  "policy_subject": "project:fabric-prod",
  "evidence_subject": "evidence:openstack:keystone:fabric-prod",
  "trust_subject": "trust:openstack:keystone:fabric-prod"
}

Keystone Role

OpenStack Keystone is the infrastructure identity root for OpenStack tenants.

In the OpenAutonomyX model, Keystone should not be treated as the final identity system. It should be bridged into the Fabric identity graph. 

Keystone identity
→ Fabric DID / VC
→ OpenFGA relation
→ OPA/AuthZEN policy decision
→ Trust score
→ Audit evidence

They allow OpenAutonomyX to express identity beyond one platform boundary:

Access Decision Flow

Actor requests action
→ Resolve actor identity
→ Resolve target construct
→ Resolve relation graph
→ Evaluate policy
→ Check trust score
→ Check approval requirement
→ Allow or deny
→ Record evidence

Deny-by-Default Rule

Access is denied unless all are true:

1. Actor identity is known. 2. Target construct is known. 3. Relation permits access. 4. Policy permits action. 5. Trust score is above threshold. 6. Required approval exists, if needed. 7. Evidence can be recorded.

Identity Evidence

Identity evidence should include:

  • Creation event
  • Verification event
  • Role assignment
  • Policy binding
  • Access grant
  • Access review
  • Revocation event
  • Incident event
  • Trust score change

Minimum Identity Gate

A construct cannot become Active unless identity binding exists.

Critical constructs require verified identity and explicit human approval.

Critical identity surfaces include:

  • Linux root/admin
  • LXD project administrator
  • Keystone administrator
  • Kubernetes cluster administrator
  • Juju controller administrator
  • Policy administrator
  • Secret administrator
  • Agent runtime administrator
  • Tool execution administrator

LXD Linux Construct Model

LXD is treated by OpenAutonomyX as the Linux-native machine fabric layer for system containers, lightweight virtual machines, and local infrastructure composition.

It sits between raw Linux and larger cloud constructs such as OpenStack or Kubernetes.

Position

Bare Metal / VPS
→ Linux
→ LXD Machine Fabric
→ OpenStack or Kubernetes
→ Juju / Operator Governance Layer
→ Fabric Services
→ Agentic OS / Agent Applications

Why LXD Matters

LXD provides a practical construct for:

  • Linux system containers
  • Lightweight virtual machines
  • Local cloud-style instances
  • Isolated development and test environments
  • Repeatable machine images
  • Edge and single-node infrastructure
  • OpenStack lab substrates
  • Kubernetes lab substrates

OpenAutonomyX Interpretation

Linux = host operating system
LXD = machine and instance fabric
OpenStack = infrastructure cloud fabric
Kubernetes = workload scheduling fabric
Juju = operator reconciliation fabric
AGenNext = agentic work fabric

LXD is not only a container tool. In this model, LXD is a local infrastructure abstraction for machines.

Construct Units

| LXD Construct | Fabric Meaning | Governance Question |
|---|---|---|
| Project | Local tenant boundary | Who owns this boundary? |
| Profile | Instance policy template | What default rules apply? |
| Image | Bootable artifact | Is image provenance trusted? |
| Instance | Runtime box | What workload and owner? |
| Network | Local communication boundary | What can connect? |
| Storage Pool | Stateful substrate | Where does data live? |
| Volume | Data box | Who owns and can restore it? |
| Snapshot | Recovery point | Is rollback governed? |
| Device | Capability attachment | What hardware/resource is exposed? |
| Remote | Federation endpoint | Is the remote trusted? |

Deployment Rule

Use LXD when the platform needs machine-level isolation without the operational weight of a full OpenStack cloud.

Use OpenStack when the platform needs multi-tenant infrastructure cloud primitives across many machines.

Use Kubernetes when the platform needs application workload scheduling.

LXD vs OpenStack vs Kubernetes

| Layer | Primary Job | Smallest Useful Unit |
|---|---|---|
| LXD | Local machine/instance fabric | Instance |
| OpenStack | Private cloud infrastructure fabric | Tenant project |
| Kubernetes | Container workload fabric | Pod |
| Juju | Operator and lifecycle fabric | Application model |
| AGenNext | Agentic work fabric | Agent / task / decision |

Recommended Single-Node Path

For a single Linux server: 

Ubuntu Server
→ LXD

→ Juju
→ Fabric Services

For current production simplicity, k0s may run directly on the host.

For clean isolation and repeatability, create LXD VMs and run k0s/OpenStack labs inside them.

LXD as OpenStack Lab Substrate

LXD can be used to create disposable machines for OpenStack testing:

This is suitable for learning, design validation, and non-production proofing.

  • Profile policy

  • Network boundaries

  • Instance lifecycle

OpenInfra Fabric Construct

This document defines the OpenAutonomyX interpretation of OpenInfra, Linux, LXD, OpenStack, Kubernetes, and Juju as one governed infrastructure fabric.

Core Thesis

OpenInfra is not only a collection of infrastructure projects.

In the OpenAutonomyX model, OpenInfra is the lower fabric of governed compute, storage, networking, identity, and lifecycle primitives. 

Linux
→ LXD
→ OpenStack
→ Kubernetes
→ Juju
→ Fabric Services
→ Agentic OS

Ordered Construct Layers

| Order | Layer | Primary Unit | Main Purpose |
|---:|---|---|---|
| 0 | Linux | Host | Operating system substrate |
| 1 | LXD | Instance | Local machine and VM fabric |
| 2 | OpenStack | Tenant / project | Private cloud infrastructure fabric |
| 3 | Kubernetes | Pod / cluster | Workload scheduling fabric |
| 4 | Juju | Model / application | Operator reconciliation fabric |
| 5 | Fabric Services | Identity / policy / graph / evidence | Governance and decision fabric |
| 6 | Agentic OS | Agent / task / workflow | Governed work execution |

Why This Order Matters

Each layer should provide a clear abstraction to the layer above it: 

Linux gives process and hardware control.
LXD gives machine boxes.
OpenStack gives cloud tenants.
Kubernetes gives workload scheduling.
Juju gives application lifecycle reconciliation.
Fabric Services give identity, policy, graph, memory, and evidence.
Agentic OS gives governed agentic work.

Do not collapse all layers into one concept.

The fabric becomes understandable when each layer owns one job.

Construct Boundary Rule

A construct is valid only when it has:

  • Identity
  • Owner
  • Purpose
  • Policy
  • Evidence
  • Lifecycle
  • Observability
  • Recovery path
  • Trust score

Minimal Path

For a single-node or early platform: 

Linux
→ k0s
→ Juju
→ Fabric Services
→ Agentic OS

LXD is added when Linux-native machine isolation is required.

OpenStack is added when multi-tenant infrastructure cloud control is required.

Full Private Cloud Path

For a full private cloud: 

Linux hosts

→ OpenStack for tenants, VMs, networks, images, volumes
→ Kubernetes clusters on top of OpenStack or bare metal
→ Juju for operators and reconciliation

OpenStack Construct Model

OpenStack is treated by OpenAutonomyX as an infrastructure fabric construct, not as the agent platform itself.

This document defines how OpenStack fits into the OpenAutonomyX / AGenNext operating model.

Position

Bare Metal / Provider Host
→ OpenStack Infrastructure Fabric
→ Kubernetes Runtime Fabric
→ Juju / Operator Governance Layer
→ Agentic OS / Fabric Services
→ Agent Applications

OpenStack belongs below Kubernetes when the platform needs to manage virtual machines, tenant networks, images, volumes, and infrastructure-level isolation.

Why OpenStack Exists in the Model

OpenStack provides:

  • Compute primitives through Nova
  • Image primitives through Glance
  • Network primitives through Neutron
  • Block storage primitives through Cinder
  • Identity primitives through Keystone
  • Object storage primitives through Swift or compatible services
  • Dashboard primitives through Horizon
  • Orchestration primitives through Heat
  • Telemetry primitives through Ceilometer/Aodh/Gnocchi where required

OpenAutonomyX Interpretation

OpenStack is not the full fabric.

OpenStack is the infrastructure substrate of the fabric. 

OpenStack = VM / network / image / volume / tenant substrate
Kubernetes = workload scheduling substrate
Juju = operator reconciliation substrate
SurrealDB / policy / identity = governance fabric substrate
AGenNext = agentic work and decision substrate

OpenStack is useful when the platform needs:

  • Multi-tenant infrastructure isolation

Policy Construct Model

Policy is the control layer that turns identity into permitted or denied action.

In the OpenAutonomyX model, policy applies across Linux, LXD, OpenStack, Kubernetes, Juju, Fabric Services, and Agentic OS.

Core Rule

No action is trusted only because an actor can technically perform it.

Every meaningful action must be authorized by policy, related identity, trust score, approval state, and evidence capture.

Policy Spine

Identity
→ Relation
→ Policy
→ Trust Score
→ Approval
→ Action
→ Evidence

Policy Layers

| Layer | Policy Surface | Example Control |
|---|---|---|
| Linux | sudoers, systemd, filesystem permissions | Who can administer the host? |
| LXD | projects, profiles, remotes, devices | Who can launch or modify instances? |
| OpenStack | Keystone roles, projects, policies | Who can create VMs, networks, volumes? |
| Kubernetes | RBAC, admission, network policy | Who can deploy workloads or read secrets? |
| Juju | controller/model permissions | Who can deploy or relate applications? |
| Fabric Services | OPA/OpenFGA/AuthZEN, graph policy | Who can access data, context, and evidence? |
| Agentic OS | tool policy, approval gates, kill switch | What may an agent do? |

Decision Model

An access decision should evaluate: 

{
  "subject": "agent:researcher-1",
  "action": "tool.execute",
  "resource": "tool:web-search",
  "context": {
    "tenant": "openautonomyx",
    "risk": "medium",
    "trust_score": 76,
    "approval_required": false
  }
}

The decision output should be explicit: 

{
  "decision": "allow",
  "reason": "subject has relation, policy permits action, trust score above threshold",
  "evidence_required": true
}

Deny-by-Default

Default state is deny.

Access is allowed only when all required checks pass:

1. Subject identity is known. 2. Resource identity is known. 3. Relation permits the action. 4. Policy permits the action. 5. Trust score is above threshold. 6. Human approval exists where required. 7. Evidence can be recorded. 8. No active suspension or kill switch applies.

Relation + Policy Split

OpenAutonomyX separates relationship checks from rule checks. 

Relation answers: does this subject have a valid relationship to the resource?
Policy answers: is this action allowed under current rules and context?

Example: 

OpenFGA-style relation:
user:alice is owner of project:fabric-prod

OPA/AuthZEN-style policy:
owner may create instance only if trust_score >= 70 and risk != critical_without_approval

Policy Risk Levels

| Risk | Meaning | Approval |
|---|---|---|
| Low | Read-only or reversible action | Usually automatic |
| Medium | Limited write or bounded execution | Policy-gated |
| High | Infrastructure, data, network, or runtime change | Human approval recommended |
| Critical | Identity, secrets, production, external effects | Human approval required |

Critical Actions

Critical actions include:

  • Creating or deleting identity roots
  • Granting administrator access
  • Reading or exporting secrets
  • Changing policy rules
  • Opening external network routes
  • Deploying production agents
  • Executing payment, commerce, or legal actions
  • Destroying infrastructure or data
  • Disabling audit logging
  • Disabling monitoring
  • Lowering trust gates

Policy Evidence

Every allow or deny decision should produce evidence:

  • Subject
  • Action
  • Resource
  • Decision
  • Reason
  • Policy version
  • Relation snapshot
  • Trust score at decision time
  • Approval reference, if applicable
  • Timestamp
  • Runtime environment

Construct Activation Gate

A construct cannot become Active unless it has:

  • Policy owner
  • Policy subject
  • Policy resource identifier
  • Minimum policy rules
  • Evidence output path
  • Deny-by-default behavior
  • Trust threshold

Governance Loop

Declare policy
→ Review policy
→ Bind policy to construct
→ Evaluate request
→ Allow or deny
→ Record evidence
→ Monitor drift
→ Update trust score
→ Reconcile access

Trust Score Calculation Model

The OpenAutonomyX Trust Score is a public, evidence-based scoring model for evaluating repositories, partners, agents, and platform services.

It is not a marketing badge. It is a governance signal generated from visible controls, review evidence, policy compliance, operational maturity, and risk posture.

Score Range

0   = no trust evidence
100 = strong public evidence, enforced controls, and active governance

Trust Formula

Trust Score =
  Security Controls        * 0.20 +
  Governance Controls      * 0.20 +
  Compliance Evidence      * 0.15 +
  Identity & Access        * 0.15 +
  Agent Safety             * 0.15 +
  Operational Reliability  * 0.10 +
  Transparency             * 0.05

Control Categories

| Category | Weight | Meaning |
|---|---:|---|
| Security Controls | 20% | Security policy, dependency review, secret scanning, vulnerability reporting, least privilege |
| Governance Controls | 20% | Review gates, CI checks, ownership, contribution policy, release discipline |
| Compliance Evidence | 15% | SOC2/GDPR-style evidence mapping, audit logs, documented controls |
| Identity & Access | 15% | DID/VC readiness, RBAC/ReBAC, access review, deny-by-default posture |
| Agent Safety | 15% | Human approval gates, kill switch, risk classification, tool permission boundaries |
| Operational Reliability | 10% | Monitoring, incident response, rollback, reproducible deployment |
| Transparency | 5% | Public docs, changelog, roadmap, known limitations, responsible disclosure |

Category Scoring

Each category is scored from 0 to 100. 

0   = missing
25  = documented but not enforced
50  = partially implemented
75  = implemented with evidence
100 = enforced, monitored, reviewed, and continuously improved

Trust Bands

| Score | Band | Meaning |
|---:|---|---|
| 0-24 | Unverified | No meaningful public evidence |
| 25-49 | Draft | Basic documentation exists, weak enforcement |
| 50-69 | Controlled | Controls exist, partial enforcement/evidence |
| 70-84 | Trusted | Strong controls, reviewable evidence, active governance |
| 85-100 | Enterprise Ready | Mature enforcement, audit evidence, operational reliability |

Minimum Gates

A project cannot be rated above Controlled unless it has:

  • SECURITY.md
  • Governance policy
  • CI checks
  • License
  • Documented maintainers or ownership model
  • Responsible disclosure process

A project cannot be rated above Trusted unless it has:

  • Evidence model
  • Access control model
  • Agent safety model, if agents or automation are involved
  • Operational reliability model
  • Review process

A project cannot be rated Enterprise Ready unless it has:

  • Continuous compliance evidence
  • Active monitoring
  • Incident response process
  • Change management process
  • Audit-ready control mapping

Example Manual Calculation

Security Controls:        70 * 0.20 = 14.00
Governance Controls:      80 * 0.20 = 16.00
Compliance Evidence:      50 * 0.15 =  7.50
Identity & Access:        60 * 0.15 =  9.00
Agent Safety:             75 * 0.15 = 11.25
Operational Reliability:  55 * 0.10 =  5.50
Transparency:             90 * 0.05 =  4.50

Total Trust Score = 67.75
Band = Controlled

Governance Rule

Trust Score must never be self-declared without evidence.

Every score requires:

1. Category inputs 2. Evidence links 3. Reviewer or automation identity 4. Timestamp 5. Version of this scoring model
